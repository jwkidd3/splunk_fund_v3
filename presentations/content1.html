What is Splunk?

Splunk is a platform for searching, monitoring, and analyzing machine-generated data

  • Universal Data Platform: Ingests any machine data
  • Real-time Analysis: Search and analyze data as it arrives
  • Scalable: Handles massive volumes of data
  • Flexible: Supports structured and unstructured data

Machine Data Types

Log Files

  • Web server logs
  • Application logs
  • Database logs
  • Security logs

Metrics & Events

  • Performance metrics
  • System events
  • Network traffic
  • IoT sensor data

Search Processing Language (SPL)

SPL: Splunk's search language for finding, filtering, and manipulating data

Basic SPL Structure:


search_terms | command1 | command2 | command3

Examples:
error | head 10
status=404 | stats count by host
index=web | where bytes > 1000 | table _time, host, bytes

SPL Search Components

Search Terms

  • Keywords: error, failed
  • Field values: status=200
  • Quoted phrases: "login failed"
  • Boolean operators: AND, OR, NOT

Commands

  • Filtering: where, search
  • Field manipulation: eval, rename
  • Statistics: stats, chart
  • Formatting: table, sort

Working with Time Ranges

Time Range: Specifies the time window for your search

Common Time Ranges

  • Last 24 hours
  • Last 7 days
  • Last 30 days
  • All time

Time Modifiers

  • earliest=-1h (1 hour ago)
  • latest=now (current time)
  • earliest=-7d@d (7 days ago at midnight)

Module 7: Search Best Practices

Goal: Fast, efficient, and accurate search results

Six fundamental best practices for optimal search performance:

  • Best Practice #1: Time is the most efficient filter
  • Best Practice #2: Specify index values at beginning
  • Best Practice #3: Include as many search terms as possible
  • Best Practice #4: Use inclusion vs exclusion effectively
  • Best Practice #5: Filter early in search pipeline
  • Best Practice #6: Use wildcards properly

Best Practice #1: Time is Most Efficient Filter

Principle: Always specify the shortest time range that meets your needs

❌ Inefficient


# Searching all time
index=main error

# Using overly broad time range
earliest=-30d latest=now error

✅ Efficient


# Specific time window
earliest=-1h latest=now index=main error

# Business hours only
earliest=-1d@8h latest=-1d@17h error

Impact: Can improve search performance by 10x or more

Best Practice #2: Specify Index Values

Principle: Always specify index names at the beginning of search

❌ Inefficient


# No index specified
error | stats count

# Index specified too late
error | where index="main"

✅ Efficient


# Index first
index=main error | stats count

# Multiple indexes if needed
(index=web OR index=app) error

Why: Reduces data set before processing begins

Best Practice #3: Include Many Search Terms

Principle: Use as many specific terms as possible early in search

❌ Inefficient


# Too generic
index=main | where status="404"

# Missing relevant terms
index=web error

✅ Efficient


# Multiple specific terms
index=main status=404 method=GET

# Include context
index=web error failed authentication

Benefit: Filters data at index level before field extraction

Best Practice #4: Inclusion vs Exclusion

Principle: Use inclusion (what you want) rather than exclusion (what you don't want)

❌ Less Efficient


# Excluding many values
index=main NOT (status=200 OR status=201)

# Using NOT extensively
index=web NOT success NOT debug

✅ More Efficient


# Include specific values
index=main (status=404 OR status=500)

# Positive matching
index=web error failed exception

!= vs NOT: != works on fields (status!=200), NOT works on text (NOT error)

Best Practice #5: Filter Early in Pipeline

Principle: Apply filters before expensive transforming commands

❌ Inefficient Order


# Filter after transformation
index=main 
| stats count by host, status 
| where status=404

# Search late in pipeline
index=web 
| eval response_time=duration/1000
| search response_time > 5

✅ Efficient Order


# Filter first
index=main status=404
| stats count by host

# Filter before transformation
index=web duration > 5000
| eval response_time=duration/1000

Pipeline Order: Search terms → Filters → Transformations → Formatting

Best Practice #6: Proper Wildcard Usage

Principle: Use wildcards strategically - they can hurt or help performance

❌ Inefficient Wildcards


# Leading wildcards (very slow)
index=main "*error"

# Too broad matching
index=web "*"

# Unnecessary wildcards
index=main "user*" (when you mean "user")

✅ Efficient Wildcards


# Trailing wildcards (faster)
index=main "error*"

# Specific patterns
index=web "user_*_login"

# Exact matches when possible
index=main "error" (instead of "error*")

Performance Impact: Leading wildcards can be 100x slower than trailing wildcards

SPL: Core Components

Understanding: Every SPL search contains up to five component types

1. Search Terms

Keywords, field=value pairs, quoted strings

error status=404 "login failed"
  • Filter events at index level
  • Most efficient component

2. Commands

Processing instructions that manipulate results

stats, eval, where, sort, table
  • Transform and format data
  • Connected by pipe (|) symbols

SPL: Advanced Components

3. Functions

Operations within commands

count(), avg(), max()
  • Statistical operations
  • Mathematical calculations

4. Arguments

Values that modify command behavior

limit=10, by host
  • Command parameters
  • Control processing

5. Clauses

Modify how commands process data

BY, AS, WHERE, OVER
  • Grouping and filtering
  • Field naming and logic

Understanding Search Pipeline Processing

Concept: Commands process data sequentially from left to right

1. Search Terms

index=main error

Filters raw events

→

2. Field Extraction

| where status!=200

Works with fields

→

3. Processing

| stats count by host

Transforms data

→

4. Formatting

| sort count

Presents results

Key Point: Each stage processes output from previous stage

Search Modes: Fast, Smart, Verbose

Purpose: Control search behavior and performance trade-offs

Fast Mode

  • Minimum field extraction
  • Best performance
  • Statistics focus
  • Default for transforming searches

Smart Mode

  • Balanced approach
  • Extracts needed fields
  • Good for mixed searches
  • Splunk decides optimization

Verbose Mode

  • Full field extraction
  • All events returned
  • Interactive exploration
  • Default for basic searches

Usage: Mode auto-selected based on search commands

Time Range Abbreviations

Efficiency: Use abbreviated time formats for faster specification

Time Units

  • s = seconds
  • m = minutes
  • h = hours
  • d = days
  • w = weeks
  • mon = months
  • y = years

Time Snap-to

  • @s = snap to second
  • @m = snap to minute
  • @h = snap to hour
  • @d = snap to day
  • @w = snap to week
  • @mon = snap to month

Examples:


earliest=-1h@h latest=@h        (Previous hour, exactly)
earliest=-7d@d latest=-1d@d     (Week ago to yesterday, midnight to midnight)
earliest=-1mon@mon latest=@mon   (Previous month, exactly)

Field Names: Case Sensitivity Rules

Important: Field names are case-sensitive, field values generally are not

❌ Case Errors


# Wrong field name case
Status=200    (should be status=200)
HOST=web01    (should be host=web01)
Clientip=*    (should be clientip=*)

# These will NOT match

✅ Correct Usage


# Correct field name case
status=200
host=web01
clientip=192.168.1.1

# Field values (usually case-insensitive)
method=GET or method=get (both work)

Tip: Use Field Sidebar to see exact field name casing

Advanced Search Techniques

Beyond Basics: Powerful techniques for complex searches

Boolean Logic


# AND is implied
index=main error failed
(same as: index=main error AND failed)

# Explicit OR
(status=404 OR status=500)

# Complex combinations
(error OR exception) AND status!=200

Field Comparisons


# Numeric comparisons
bytes > 1000
duration <= 5.0
response_time >= avg_time

# Field to field
src_bytes > dest_bytes

Fields: Searchable name-value pairs extracted from your data

Default Fields

  • _time - Event timestamp

  • sourcetype - Data format
  • index - Index name

Extracted Fields

  • Automatically discovered

Essential Splunk Commands

Commands: Tools to manipulate, format, and analyze search results

Data Selection

  • fields - Select specific fields
  • table - Display tabular format
  • dedup - Remove duplicates

Data Transformation

  • rename - Change field names
  • eval - Create calculated fields
  • sort - Order results

Data Selection Commands

fields Command

Include or exclude specific fields


# Include fields
| fields host, status, _time

# Exclude fields
| fields - _raw, _indextime

table Command

Display results in tabular format


# Organize columns
| table _time, host, status

# Order matters
| table status, host, _time

Performance Tip: Use fields command early in pipeline

Data Transformation Commands

rename Command

Change field names for clarity


# Single field
| rename clientip as "Client IP"

# Multiple fields
| rename clientip as IP, status as Code

dedup Command

Remove duplicate records


# By single field
| dedup clientip

# By multiple fields
| dedup clientip, useragent

Best Practice: Use rename for business-friendly field names

Search Pipeline Optimization

Goal: Build efficient, readable search pipelines

Command Ordering

  1. Filter early (search terms)
  2. Remove duplicates (dedup)
  3. Select fields (fields)
  4. Transform data (eval, rename)
  5. Format output (table, sort)

Performance Tips

  • Minimize data early
  • Use specific search terms
  • Limit unnecessary fields
  • Avoid expensive operations late

Search Troubleshooting

Quick Solutions: Common problems and fixes

No Results?

  • Expand time range
  • Check field name casing
  • Try wildcards (host=web*)

Slow Performance?

  • Narrow time range
  • Add index specification
  • Filter early in pipeline
