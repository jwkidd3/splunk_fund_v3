Making SPL Pipelines More Readable

Best Practice: Format search pipelines for better readability

❌ Hard to Read


index=web sourcetype=access_* status=503 | stats sum(price) as lost_revenue | eval lost_revenue = "$" + tostring(lost_revenue, "commas")

✅ Well Formatted


index=web sourcetype=access_* status=503 
| stats sum(price) as lost_revenue 
| eval lost_revenue = "$" + tostring(lost_revenue, "commas")

Tip: Enable auto-formatting in Preferences > SPL Editor

Transforming Commands

Purpose: Transform search results into statistical data tables

Frequency Analysis

  • top - Most common values
  • rare - Least common values

Statistical Analysis

  • stats - Calculate statistics
  • chart - Create data series
  • timechart - Time-based analysis

The top Command

Purpose: Find the most common values of a given field

Basic Usage:


# Default: top 10 values with count and percent
index=security sourcetype=linux_secure (fail* OR invalid)
| top src_ip

# Limit results and remove percentages
index=main sourcetype=access_combined status=200 file=success.do
| top productId limit=5 showperc=false

# Custom count field name
index=network sourcetype=cisco_wsa_squid
| top cs_username x_webcat_code_full limit=3 
  countfield="Total Viewed" showperc=f

The rare Command

Purpose: Find the least common values of a given field

Security Analysis Example:


# Find rarely accessed files (potential security threats)
index=main sourcetype=access_combined status=200
| rare file

# Break down by time period
index=main sourcetype=access_combined status=200
| rare file by date_month

# Limit to single rarest value
index=sales sourcetype=vendor_sales
| rare product_name showperc=f limit=1

Use Case: Identify suspicious file access patterns or unusual behavior

The stats Command

Purpose: Calculate statistics on data matching your search criteria

Common Functions

  • count - Number of events
  • dc(field) - Distinct count
  • sum(field) - Sum of values
  • avg(field) - Average value

Advanced Functions

  • list(field) - All values
  • values(field) - Unique values
  • min(field) - Minimum value
  • max(field) - Maximum value

stats count

Purpose: Count events matching search criteria

Examples:


# Simple count
index=security sourcetype=linux_secure (invalid OR failed)
| stats count

# Count with custom name
index=security sourcetype=linux_secure (invalid OR failed)
| stats count as "Potential Issues"

# Count by field
index=main sourcetype=access_combined (file=cart.do OR file=success.do) status=200
| stats count as Transactions by file
| rename file as Function

stats dc() - Distinct Count

Purpose: Count unique values for a given field

User Session Analysis:


# Count unique sessions
index=main sourcetype=access_combined
| stats dc(JSESSIONID) as Logins

# Break down by IP address
index=main sourcetype=access_combined
| stats dc(JSESSIONID) as Logins by clientip
| sort -Logins

# Multiple metrics
index=network sourcetype=cisco_wsa_squid
| stats dc(s_hostname) as "Websites visited:"

stats sum() - Calculate Totals

Purpose: Sum numeric values for analysis

Bandwidth and Revenue Analysis:


# Total bandwidth consumption
index=main sourcetype=access_combined status=200
| stats sum(bytes) as TotalBytes by file
| sort file

# Revenue calculation with multiple functions
index=sales sourcetype=vendor_sales
| stats count(price) as "Units Sold",
        sum(price) as "Total Sales" by product_name
| sort -"Total Sales"

stats avg() - Performance Analysis

Purpose: Calculate average values for performance monitoring

Database Performance Example:


# Average query time
index=main sourcetype=db_audit
| stats avg(Duration)

# Performance by command type
index=main sourcetype=db_audit
| stats avg(Duration) as "time to complete" by Command
| sort -"time to complete"

# Network performance
index=network sourcetype=cisco_wsa_squid
| stats avg(sc_bytes) as "Average Bytes" by usage

Note: Events without the field or with invalid values are excluded

stats list() vs values()

Key Difference: list() shows all values, values() shows unique values only

list() - All Values


# Shows all values (including duplicates)
index=network sourcetype=cisco_wsa_squid
| stats list(s_hostname) as "Websites visited:" by cs_username

# Browser usage with counts
index=main sourcetype=access_combined
| stats list(useragent) as "All Agents",
        count as "Times used" by useragent

values() - Unique Only


# Shows unique values only
index=security sourcetype=linux_secure fail*
| stats values(user) as "User Names",
        count(user) as Attempts by src_ip

# Clean browser list
index=main sourcetype=access_combined
| stats values(useragent) as "Agents used"

What Are Reports?

Reports: Saved searches that can show events, statistics, or visualizations

  • Reusable: Run the same analysis repeatedly
  • Fresh Data: Returns current results each time
  • Interactive: Enable drilldown to underlying events
  • Shareable: Distribute across teams and dashboards
  • Scheduled: Can run automatically

Best Practice: Use descriptive naming conventions like `Sales_Report_QuarterlySalesRevenue`

What Is a Dashboard?

  • Interactive: Enable drilldown and filtering

Efficiency Tip: Create panels from reports for reusability and consistency

What Is a Lookup?

Lookup: Add fields from external data sources to enrich your events

Common Use Cases

  • Product catalogs (ID to name/price)
  • User directories (ID to username)
  • Status code descriptions
  • Geographic information
  • Asset inventories

Benefits

  • Add business context
  • Enrich technical data
  • Enable meaningful reports
  • Improve user experience
  • Support compliance needs

Lookup File Structure

CSV Format: First row contains field names (headers)

Example: products.csv


productId,product_name,categoryId,price,sale_price,Code
DB-SG-G01,Mediocre Kingdoms,STRATEGY,24.99,19.99,A
DC-SG-G02,Dream Crusher,STRATEGY,39.99,24.99,B
FS-SG-G03,Final Sequel,STRATEGY,24.99,16.99,C
WC-SH-G04,World of Cheese,SHOOTER,24.99,19.99,D
WC-SH-T02,World of Cheese Tee,TEE,9.99,6.99,E
  • Input Field: productId (matches field in events)
  • Output Fields: product_name, price, etc. (added to events)
  • Case Sensitive: Field values are case-sensitive by default

Lookup Commands

inputlookup

View lookup file contents


# Review lookup data
| inputlookup products.csv

# Validate lookup definition
| inputlookup products_lookup

lookup

Apply lookup to search results


# Manual lookup application
index=main sourcetype=access_combined
| lookup products_lookup 
    productId as productId 
    OUTPUT product_name as ProductName

# Multiple outputs
| lookup products_lookup productId 
  OUTPUT product_name, price

Automatic Lookups

Seamless Integration: Fields automatically added without lookup command

Configuration Steps:


1. Settings → Lookups → Automatic lookups
2. Create new automatic lookup:
   - Name: products_auto_lookup
   - Lookup table: products_lookup
   - Apply to: sourcetype = access_combined_wcookie
   - Input fields: productId = productId
   - Output fields: product_name = ProductName, price = Price

3. Use enriched fields in searches:
index=main sourcetype=access_combined file=success.do status=200
| stats sum(Price) as Revenue by ProductName
| sort -Revenue

Advanced Lookup Features

Time-Based Lookups

  • Match based on time ranges
  • Historical pricing data
  • Seasonal adjustments
  • Dynamic policy enforcement

Lookup Options

  • local=true - Search head specific
  • update=true - Overwrite fields
  • Case-insensitive matching
  • Default values for misses

Complex Lookup Example:


# Multi-field lookup with time component
| lookup seasonal_pricing 
    productId as productId 
    _time as sale_date 
    OUTPUT adjusted_price, promotion_code

Why Scheduled Reports?

Automation: Deliver insights automatically without manual intervention

  • Executive Reporting: Monthly, weekly, daily management roll-up reports
  • Performance Optimization: Dashboard performance through scheduled summary indexing
  • Distribution: Automatically sending reports via email to stakeholders
  • Compliance: Regular compliance and audit reporting
  • Operational Intelligence: Regular system health and performance reports

Business Value: Reduce manual reporting effort and ensure consistent delivery

Creating a Scheduled Report - Define Schedule

Configuration Options: Flexible scheduling to meet business needs

Schedule Frequencies

  • Run every hour: Real-time operational monitoring
  • Run every day: Daily business summaries
  • Run every week: Weekly performance reviews
  • Run every month: Monthly executive reports
  • Run on Cron Schedule: Custom timing patterns

Schedule Configuration

  • Time Range: Data window for each run
  • Schedule Window: Execution time tolerance
  • Priority: Higher priority for critical reports
  • Concurrent Searches: Resource management

Scheduled Report Actions

Trigger Actions: What happens when the report runs

Data Actions

  • Log Event: Create indexed, searchable log event
  • Output to Lookup: Send results to CSV lookup file
  • Output to Telemetry: Send metrics to Splunk (opt-in)

Notification Actions

  • Send Email: Email results to recipients
  • Run Script: Execute custom scripts
  • Webhook: HTTP POST to specified URL

Report Embedding

Web Integration: Embed reports directly in web pages

Embedding Process:


1. Navigate to Reports → Select your report
2. Click Edit → Embed
3. Enable Report Embedding
4. Copy the provided iframe code
5. Paste into your HTML-based webpage

Requirements:
- Report must be scheduled first
- Embedded report updates with scheduled runs
- Viewers need appropriate Splunk permissions

Use Cases: Corporate intranets, executive dashboards, operational centers

What Are Alerts?

Proactive Monitoring: Automated notification when conditions are met

Alert Types

  • Scheduled Alerts: Run at regular intervals
  • Real-time Alerts: Monitor continuously

Alert Actions

  • Create entry in Triggered Alerts
  • Log an event to Splunk
  • Output results to lookup file
  • Send email notifications
  • Execute webhooks
  • Perform custom actions

Choosing Real-time vs Scheduled Alerts

Decision Factors: Urgency, resource usage, and business requirements

Scheduled Alerts

  • Operation: Search runs at defined interval
  • Evaluation: Trigger condition checked when search completes
  • Resource Usage: Lower system impact
  • Use Cases: Daily summaries, trend analysis, compliance
  • Latency: Delayed by schedule interval

Real-time Alerts

  • Operation: Search runs constantly in background
  • Evaluation: Conditions evaluated within time window
  • Resource Usage: Higher system impact
  • Use Cases: Security incidents, system failures
  • Latency: Near real-time notification

Setting Up Scheduled Alerts

Configuration Steps: Define schedule, conditions, and actions

Frequency Options:


• Run every hour: 0 * * * * (every hour on the hour)
• Run every day: 0 6 * * * (daily at 6:00 AM)
• Run every week: 0 6 * * 1 (weekly on Monday at 6:00 AM)
• Run every month: 0 6 1 * * (monthly on 1st day at 6:00 AM)
• Custom Cron Schedule: Define complex patterns

Time Range Examples:
• Last 24 hours: -24h@h to @h
• Previous complete day: -1d@d to @d
• Last 7 days: -7d@d to @d
• Previous week: -1w@w1 to @w1

Setting Trigger Conditions

Condition Types: Define when alerts should fire

Scheduled Conditions

  • Number of Results: Alert when result count exceeds threshold
  • Number of Hosts: Alert when unique hosts exceed limit
  • Number of Sources: Alert when unique sources exceed limit
  • Custom Condition: Use SPL for complex logic

Real-time Conditions

  • Per-Result: Trigger on each matching result
  • Rolling Window: Evaluate conditions over time periods
  • Threshold Based: Count, hosts, sources thresholds
  • Custom Logic: Complex SPL-based conditions

Alert Action Configuration

Execution Modes: How often actions fire when conditions are met

Once (Recommended)

  • Executes actions one time for all matching events
  • Prevents alert flooding
  • Example: 40 results → 1 alert
  • Best for: Summary alerts, status updates

For Each Result

  • Executes actions for each matching result
  • Can cause alert storms
  • Example: 40 results → 40 alerts
  • Best for: Individual incident tracking

Throttling: Use to suppress duplicate alerts within specified time ranges

Alert Severity and Management

Organization: Classify and prioritize alerts effectively

Severity Levels

  • Info: Informational notifications
  • Low: Minor issues requiring attention
  • Medium: Significant events needing response
  • High: Critical issues requiring immediate action
  • Critical: Emergency situations

Alert Management

  • Triggered Alerts: View via Activity menu
  • Alert History: Track alert firing patterns
  • Alert Editing: Modify conditions and actions
  • Alert Suppression: Temporarily disable alerts

Email Alert Configuration

Professional Notifications: Customize email content and formatting

Email Configuration Options:


To: recipient@company.com, team@company.com
Priority: Normal, Low, High, Highest
Subject: Splunk Alert: $name$ (uses alert title)
Message: Custom message with context and instructions

Include Options:
☑ Link to Alert (direct access to alert definition)
☑ Link to Results (view triggering events)
☑ Search String (see the search that triggered)
☑ Trigger Condition (understand why it fired)
☑ Trigger Time (when alert was triggered)
☐ Inline Table (results embedded in email)
☐ Attach CSV (results as attachment)
☐ Attach PDF (formatted report attachment)

Format Options:
• HTML & Plain Text: Rich formatting with fallback
• Plain Text: Simple text-only emails

Alert Best Practices

Effective Alerting: Minimize noise while maximizing actionability

Design Principles

  • Actionable: Every alert should require action
  • Specific: Clear, descriptive alert names
  • Contextualized: Include relevant details
  • Escalated: Route to appropriate personnel
  • Tracked: Monitor alert effectiveness

Common Pitfalls

  • Alert Fatigue: Too many non-critical alerts
  • False Positives: Poorly tuned thresholds
  • Missing Context: Alerts without explanation
  • No Follow-up: Alerts without response tracking
  • Over-alerting: Multiple alerts for same issue
